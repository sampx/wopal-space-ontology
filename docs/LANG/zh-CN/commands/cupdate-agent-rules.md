# 创建或更新 Agent 规则

创建或更新项目级或目录级 `AGENTS.md`。

**用户输入参数**: `$1` `$2`

**参数说明**: `[路径|项目名] [额外规则上下文]`。仅给项目名时，去 `projects/` 查目录匹配推断路径。路径或项目名必输其一, 未提供则向用户确认.

---

## 核心原则

### 文档路径

- 项目级：`<project>/AGENTS.md`
- 目录级：`<target-directory>/AGENTS.md`
- 若用户偏好语言非英文，先在同目录生成 `AGENTS.<locale>.md` 供用户审核；确认通过后，再更新 `AGENTS.md`

### 目的

`AGENTS.md` 是给 coding agent 的开发规范。

它只回答三件事：

1. 这个项目是什么。
2. 代码结构在哪里。
3. 开发、测试、验证时必须遵守哪些项目技术规则。

它不是 README，不是 DESIGN，也不是业务规则文档。

### 前置条件

生成前应优先读取：

- 目标 `AGENTS.md`
- 最近的上级 `AGENTS.md`
- `.wopal-space/STRUCTURE.md`
- 相关 PRD
- 相关 DESIGN
- 相关 `BUSINESS_RULES.md`
- 项目 package / build / test / typecheck / lint 配置
- 目标范围关键源码文件
- `rules-context`（如有）

WopalSpace 常见文档位置：

```text
projects/<product>/docs/PRD.md
projects/<product>/docs/PRD-*.md
projects/<product>/docs/DESIGN.md
projects/<product>/docs/DESIGN-*.md
projects/<product>/docs/BUSINESS_RULES.md
<project>/AGENTS.md
```

项目推断规则：

- 若 `$1` 是明确路径，直接使用
- 若 `$1` 是项目名，先按 `.wopal-space/STRUCTURE.md` 中的项目结构定位候选项目
- 若命中唯一项目，先向用户确认“是否指这个项目路径”
- 若存在多个同名/近名候选，列出候选并让用户选择

### 输出语言与文件命名

- 若用户偏好语言不是英文，必须先生成用户偏好语言版本供用户审核；用户确认通过后，再翻译并更新正式英文版。
- 用户偏好语言版本命名为 `AGENTS.<locale>.md`，例如 `AGENTS.zh-CN.md`。
- `<locale>` 必须使用 IETF BCP 47 / RFC 5646 标记。
- 正式英文版保持无后缀命名：`AGENTS.md`。
- 若用户偏好语言为英文，则直接生成或更新 `AGENTS.md`，不生成 `AGENTS.en-US.md` 等英文变体文件。

### 核心规则

- 项目级文档写项目边界；目录级文档只写该目录直接负责的规则，不复制整份项目规范。
- 正文只写项目特定技术实施规则；产品意图、设计细节、业务行为以权威文档引用为准。
- 从 PRD 提炼：一句话项目定位，以及只影响实现的范围约束。
- 从 DESIGN 提炼：精简执行链、目录 / 模块职责、技术栈选择、接口 / 状态 / 配置 / 输出 / 错误处理契约。
- BUSINESS_RULES.md 只作为权威文档链接引用，禁止从中提取任何规则写入 AGENTS.md 正文。
- 从代码 / 配置提炼：build、test、typecheck、lint、format 命令，基本开发命令，已使用框架 / 库约束，本地实现约定。
- 技术栈只在架构节声明一次；实施规则节不重复。
- `rules-context` 是用户直接要求补充的项目技术规则，应直接并入对应章节；若内容明显属于 PRD、业务规则或临时计划，则不写入 `AGENTS.md`，并在完成后响应中说明。

### 写作质量线

- `AGENTS.md` 不超过 300 行；超出时压缩内容、改为引用，或拆到更近的子目录 `AGENTS.md`。
- 必须保留基本开发 / 测试命令和适用的验证要求。
- 只写实施、测试、验证相关规则。
- 优先引用，不重复解释。
- 使用直接、可执行的祈使句。
- 使用当前实现事实，不写路线猜测。
- 明确边界：本范围负责什么，不应改什么。
- 用户偏好语言版本的所有标题使用目标语言；正式英文版标题保持英文。
- 禁止事项只放项目特定开发禁忌；不写“不要复述 PRD”这类维护性废话。

禁止：

- README 式项目介绍
- “本文件适用于……”这类低信息章节
- PRD 愿景、用户叙事、路线
- 业务规则复述
- 大段 DESIGN 原文
- 架构大图、目录百科、API / 命令清单
- backlog / task plan / 命令记录等临时实施产物链接（除非用户明确要求）

## 更新模式

更新既有 `AGENTS.md` 时：

1. 保留准确的项目特定开发 / 测试规则。
2. 删除过时、重复、冗长和通用空间规则。
3. 把复制的 PRD / DESIGN 长文改成引用 + 精简技术约束；删除任何从 BUSINESS_RULES.md 搬来的规则。
4. 补齐一句话项目定位、精简架构、目录职责、开发测试命令。
5. 将 `rules-context` 直接并入对应章节；若有越界内容，在完成后响应中说明未写入原因。
6. 控制在 300 行以内。
7. ⚠️ **规则与规范初次创建后不可直接修改。** 更新既有 `AGENTS.md` 时，永远不能直接添加、修改或删除任何已有规则 / 规范内容。只能推荐变更建议；所有增删改必须经用户同意后才能执行。

## 确认策略

正式写入或覆盖 `AGENTS.md` 前，必须先展示完整优化方案并获得用户明确确认。

方案至少包含：

1. 目标文件路径
2. 将引用的权威文档
3. 一句话项目定位
4. 拟保留 / 新增 / 删除 / 压缩的规则摘要
5. 架构 / 目录说明方案
6. 开发、测试、验证要求
7. `rules-context` 的并入位置
8. 若可能超过 300 行，对应的压缩或拆分方案

未确认前，不得写入、覆盖或重排正式英文版 `AGENTS.md`。

若已先生成用户偏好语言版本，用户确认后：

1. 先更新用户偏好语言版本。
2. 再翻译并更新对应英文正式版。
3. 英文正式版必须与已确认版本语义一致。

## 质量检查

- [ ] 目标路径明确或可安全推断
- [ ] 已参考目标和上级 `AGENTS.md`（如存在）
- [ ] 已引用 PRD、DESIGN、`BUSINESS_RULES.md`（如存在）
- [ ] `AGENTS.md` 不超过 300 行
- [ ] 保留基本开发 / 测试命令说明
- [ ] 规则聚焦技术实施、测试、验证
- [ ] 若用户偏好语言非英文，已先生成用户偏好语言版本
- [ ] 引用权威文档，而不是复制长文
- [ ] 未从 BUSINESS_RULES.md 提取任何规则写入正文
- [ ] 测试节包含 TDD 要求
- [ ] 用户偏好语言版本标题使用目标语言
- [ ] 禁止事项只有项目特定开发禁忌，无维护性废话
- [ ] 写入前已展示完整优化方案并获得用户确认
- [ ] 用户确认后已同步更新正式英文版（如适用）

## 完成后响应

用用户语言回复：

1. 更新的文件路径
2. 覆盖范围
3. 关键新增 / 修改规则
4. 被忽略的 `rules-context` 内容（如有）及原因
5. 缺失的权威文档引用或假设
